=========================================
AEGIS SECURITY SOLUTIONS: ENCRYPTION AND CRYPTOGRAPHIC CONTROL POLICY
=========================================
Document ID: SEC-ENC-012-2026
Effective Date: March 1, 2026
Owner: Enterprise Architecture & Security (EAS)
Classification: CONFIDENTIAL / INTERNAL USE ONLY

1. PURPOSE
The purpose of this Encryption Policy is to define the minimum cryptographic standards required to protect Aegis Security Solutions's ("Aegis") sensitive information (Confidential, Restricted, and Internal use) across its lifecycle, encompassing data traversing networks (Data in Transit) and data residing on physical or logical media (Data at Rest).

2. SCOPE
This policy applies comprehensively to all electronic data processed, transmitted, stored, or managed by Aegis employees, contractors, third-party delegates, on-premises systems, cloud environments (IaaS, PaaS, SaaS), mobile devices, and vendor infrastructure processing corporate data.

3. REQUIRED CRYPTOGRAPHIC STANDARDS
Aegis mandates adherence to NIST FIPS 140-2 (or newer FIPS 140-3) validated cryptographic modules. Proprietary or custom-developed encryption algorithms are expressly strictly prohibited across all Aegis environments.

3.1. Data at Rest (DAR)
All Aegis Restricted and Confidential data must be encrypted at rest.
- Block-Level & Volume Encryption: Servers, laptops, workstations, and mobile devices must utilize Full Disk Encryption (FDE) such as BitLocker (Windows), FileVault 2 (macOS), or LUKS (Linux) using AES-256 or higher (XTS-AES-256).
- Database Encryption: Databases housing Personally Identifiable Information (PII) or Protected Health Information (PHI) must employ Transparent Data Encryption (TDE) or column-level encryption utilizing AES-256 (GCM or CBC mode with random IVs).
- Cloud Storage: Amazon S3 buckets, Azure Blob Storage, and GCP Cloud Storage must enable default encryption (SSE-S3, SSE-KMS, or Customer Managed Keys (CMK) via AWS KMS). Unencrypted cloud storage provisions will be automatically remediated by the Cloud Security Posture Management (CSPM) tool.

3.2. Data in Transit (DIT)
All network communications that transit untrusted networks (e.g., the internet) or transmit sensitive data within internal corporate boundaries (e.g., east-west traffic between microservices) must be encrypted.
- Transport Protocol: TLS version 1.3 is the mandated standard for all new deployments. TLS 1.2 is permitted exclusively configured to support strong cipher suites with Forward Secrecy (e.g., ECDHE-RSA-AES256-GCM-SHA384).
- Deprecated Protocols: SSL v2/v3, TLS 1.0, and TLS 1.1 are explicitly forbidden. Internal load balancers must actively reject handshake attempts leveraging deprecated protocols.
- SSH and VPN: Administrative connections to network devices and servers must utilize SSHv2 (using Ed25519 or RSA-4096 keys). VPN tunnels must leverage IPSec (IKEv2) utilizing AES-256-GCM or WireGuard.

3.3. Key Management Lifecycle
Cryptographic keys must be strictly governed and managed using enterprise Key Management Systems (KMS) or Hardware Security Modules (HSMs) operating at minimum FIPS 140-2 Level 3 certification.
- Key Generation: Keys must be generated using approved Random Number Generators (RNG) (e.g., SP 800-90A). 
- Key Storage: Private and secret keys must remain protected and never hardcoded in source code, configuration files, or unencrypted repositories. 
- Key Rotation: Symmetric encryption keys must be rotated annually (or proactively rotated if personnel change or key compromise is suspected). Asymmetric keys (certificates) must not exceed a validity period of 398 days.

4. COMPLIANCE AND ENFORCEMENT
Violations will result in disciplinary action up to and including termination of employment or revocation of vendor contracts. Continuous monitoring via SIEM and CSPM will track encryption posture. Exceptions must formally undergo CISO-approved risk acceptance lasting no longer than 6 months.
